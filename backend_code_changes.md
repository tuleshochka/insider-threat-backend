# План изменений бэкенда (соответствие ТЗ из Главы 3)

Так как обучение моделей происходит в облаке (в ноутбуке `insider_threat_training_r52.ipynb`), код бэкенда должен быть сфокусирован **исключительно на быстром и надежном инференсе** (предсказании) и обработке данных.

В связи с новыми требованиями ТЗ (разделы 3.2, 3.3, 3.4) в кодовую базу необходимо внести следующие изменения:

## 1. Очистка кодовой базы от лишнего (Рефакторинг)
Поскольку модель обучается в Colab, скрипты локального обучения в бэкенде больше не нужны. Они только создают путаницу.
* **Что сделать:** Удалить файлы `train_multi_input.py`, `two_level_pipeline.py`, `supervised_benchmark.py` и `user_level_classifier.py` из корня бэкенда.
* **Обоснование:** Бэкенд должен быть легковесным и содержать только FastAPI, сервисы предобработки и скрипты инференса (загрузка весов `.pt` и прогон данных).

## 2. Изменения в ноутбуке обучения (`insider_threat_training_r52.ipynb`)
* **Требование ТЗ (3.2):** *«Ключевым требованием... выступает достижение целевой метрики F2-меры на уровне не ниже 0,8»*.
* **Что сделать:** В ячейках, где происходит подбор оптимального порога классификации (threshold), изменить функцию оценки с `f1_score` (или индекса Юдена) на **F2-меру** (`fbeta_score(beta=2)`). Выгружаемый в бэкенд порог отсечения должен быть оптимизирован именно под полноту (Recall).

## 3. Математическая интерпретируемость (SHAP)
* **Требование ТЗ (3.2):** *«Модуль должен генерировать формализованное объяснение... с выделением признаков. Числовое представление влияния отдельных факторов...»*
* **Что сделать:** В сервисе инференса (`app/services/` или `app/api/`) при генерации ответа об инциденте необходимо вычислять SHAP-значения (или интегрированные градиенты) для конкретного вектора. API должен отдавать JSON, где помимо `anomaly_score: 0.95` будет массив конкретных триггеров: `{"after_hours_logons": +0.35, "email_size": +0.12}`. Механизма Attention, встроенного в нейросеть, недостаточно для детального помодульного отчета.

## 4. Локальное кэширование при отказе БД (Отказоустойчивость)
* **Требование ТЗ (3.3):** *«В случае кратковременного отказа системы хранения данных... модули должны кэшировать поступающие события локально...»*
* **Что сделать:** В FastAPI эндпоинте приема событий (вероятно, `POST /events` или `POST /predict`) обернуть сохранение в PostgreSQL (через SQLAlchemy) в блок `try/except`. При падении БД с ошибкой подключения (ConnectionError) сохранять вектор в локальный буфер (например, в SQLite-таблицу `fallback_queue` или JSONL-файл). Добавить фоновую задачу (background task), которая раз в минуту пытается протолкнуть эти данные в основную СУБД.

## 5. Жесткий лимит инференса в 5 секунд (Производительность)
* **Требование ТЗ (3.2):** *«время инференса модели не должно превышать пяти секунд...»*
* **Что сделать:** Убедиться, что `FastAPI` эндпоинт полностью асинхронный (`async def`). Вызов модели PyTorch и расчет SHAP — это тяжелые блокирующие (CPU-bound) операции. Их нужно обязательно вызывать через `await run_in_threadpool(model.predict, data)`, чтобы не блокировать весь веб-сервер при обработке запроса.

## 6. Обработка пропусков в данных (Предобработка)
* **Требование ТЗ (3.4):** *«Допускается наличие пропусков... не превышает 5%; такие записи требуется восстанавливать методами интерполяции»*.
* **Что сделать:** В классе `DataPreprocessor` (в `app/services/`) добавить логику: если в суточном векторе пропущено <5% данных, использовать линейную интерполяцию (`.interpolate()`). Если больше — отклонять вектор или генерировать алерт о битой телеметрии. Сейчас пропуски часто просто заполняются нулями.
